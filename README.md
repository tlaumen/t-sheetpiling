# T-sheetpiling - an interactive sheet pile solver

## Run

    uv run server/backend.py

Then open http://127.0.0.1:8765/ in a browser. The backend auto-detects
which shared library file to load (.so / .dylib / .dll) based on what's
present in bin/ -- build that first, see below.

## Build (Linux / macOS)

    make

This builds two things into bin/:
- bin/sheetpile        -- CLI binary (src/main.c), useful for quick sanity checks
- bin/libsheetpile.so   -- shared library the web backend calls via ctypes
  (bin/libsheetpile.dylib on macOS)

`make lib` builds only the shared library; `make run` builds and runs the CLI.

## Layout

- include/, src/ -- the C solver: soil spring law (tri-linear
  active/neutral/passive), beam FE assembly, dense linear solver,
  Newton-Raphson driver, and api.c (flat FFI entry point for ctypes).
- server/backend.py -- stdlib-only Python HTTP server. Serves the static
  frontend and a POST /solve endpoint that calls the shared library
  in-process via ctypes (no subprocess spawn -- this is what keeps
  round-trip latency under ~1ms).
- web/ -- static frontend: index.html (two tabs), style.css, app.js
  (vanilla JS, no build step, no frameworks).
- bin/ -- build output only (sheetpile, libsheetpile.*). Not committed;
  regenerated by `make`.

## Build (Windows, cross-compiled from Arch Linux)

Install the cross-compiler:

    sudo pacman -S mingw-w64-gcc

Build the DLL:

    x86_64-w64-mingw32-gcc -std=c11 -O2 -Iinclude -shared -static-libgcc -o bin/libsheetpile.dll \
        src/soil_springs.c src/beam_fe.c src/linalg.c src/solver.c src/api.c -lm

Copy the resulting `bin/libsheetpile.dll` to the Windows machine alongside
the rest of the project. Colleagues just need Python and `uv` there --
no compiler or `make` required on their end.
